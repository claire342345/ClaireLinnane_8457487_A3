SQL Injection








Source: https://www.linkedin.com/pulse/understanding-sql-injection-software-qa-engineers-mejbaur-bahar-fagun/ 		Often abbreviated as SQLi, this form of cyber-attack works by inserting malicious code into the query's input form thereby enabling the hacker to gain access of the database and its contents. SQL injections commonly target the WHERE clause of a SELECT query, however they can also occur in:
  • UPDATE statements or the WHERE clause
  • INSERT statements
  • SELECT statements, within the table or column name or within the ORDER BY clause
SQLi attacks typically fall into 1 of 3 categories: In-band SQLi, the most common type of attack, Inferential SQLi otherwise known as blind SQLi, and Out-of-Band SQLi, the least common type of attack. In-band SQLi describes when attackers inject malicious code directly into the application's communication channel, allowing them to retrieve data through the same pathway. Inferential SQLi involves deducing information indirectly from the application's responses without directly communicating with the database. Lastly, out-of-Band SQLi utilises alternative communication channels like HTTP requests to extract data from the database, providing attackers with different avenues to exploit vulnerabilities. SQL injection can be prevented by using parameterised queries or prepared statements to separate SQL code from user input, thus preventing malicious injection attacks.

Source: https://www.crowdstrike.com/cybersecurity-101/sql-injection/ | https://portswigger.net/web-security/sql-injection
Cross-Site Scripting (XSS)




Source: https://medium.com/@suhaslingam/all-about-cross-site-scripting-xss-part-1-reflected-xss-c70d77fabf99 		Cross-site scripting/ XSS attacks occur when a user visits a web page or application containing malicious code. An XSS attack begins with the attackers identifying a vulnerability in a web application like a search box or a comment form that is not properly validated. The attacker then injects a malicious script into the vulnerable input field and when the user interacts with the contaminated content the malicious script executes proceeding to compromise the user’s browser or even PC. XSS attacks come in 3 forms: stored XSS, reflected XSS and Document Object Model (DOM)-based XSS. Malicious script that has been injected into a website and executes when accessed by other users is stored XSS while injected script that is reflected off a web server and executed in the user's browser is Reflected XSS. Lastly, DOM-nased XSS exploits weaknesses within the Document Object Model (DOM) of a webpage to run harmful scripts in the user's browser. XXS attacks can be prevented by properly thoroughly validating user input, and employing strategies like the Content Security Policy (CSP) to restrict the execution of untrusted scripts.


Source: https://www.cloudflare.com/learning/security/threats/cross-site-scripting/ | https://chat.openai.com/c/9604c479-d710-4deb-9d6c-92345fd347cb
Cross-Site Request Forgery (CSRF)





Source: https://securitysouls.com/a-tale-of-cross-site-request-forgery-csrf/ 		The key aspect of this attack is forgery: CSRF operates by forging a request on behalf of the victim, deceiving the browser into sending an unauthorized request by an authenticated user to a web application. A successful CSRF attack relies on three conditions being met: a relevant action to induce, an application that relies solely on session cookies for user identification in HTTP requests, and no unpredictable request parameters. That being said, there are several anti-CSRF measures that can be deployed by both the target website and of the victim's browser such as CSRF tokens (unique and random values generated by the server-side application), SameSite cookies (browser-based restriction of cookies in cross-site requests), and Referer-based validation (verification of request origin from the application's domain). CSRF can be prevented through the use of anti-CSRF tokens, unique tokens generated for each user session and included in requests to validate the origin of the request and prevent unauthorised actions.

Distributed Denial of Service (DDoS)



Source: https://bunny.net/academy/security/what-are-distributed-denial-of-service-ddos-attacks/ 		DDoS attacks capitalise on the capacity limits of a website by harnessing the collective power of multiple online devices, forming a botnet, and inundating the target website with artificial traffic, thus preventing legitimate users from accessing the website or server. There are several types of DDoS attacks and they identified by the extent to which they impact the network. Volumetric Attacks strives to control as much of the user and internet’s bandwidth as possible, while protocol attacks aim to consume all available capacity of web servers or other resources, such as firewalls. Lastly, application-layer attacks act to overwhelm the target's resources. This type of cyber-attack is frequently employed by hacktivists or extortionists to disrupt services or coerce targets into meeting their demands. The prevalence of DDoS attacks can be reduced through the implementation of network traffic filtering, rate limiting, and deploying DDoS protection services that would help prophylactically identify and block malicious traffic before it overwhelms the system.


Source: https://www.imperva.com/learn/ddos/denial-of-service/ | https://www.fortinet.com/resources/cyberglossary/ddos-attack
Man-in-the-Middle (MitM)



Source: https://bunny.net/academy/security/what-is-man-in-the-middle-mitm-attack/ 		An MitM attack, also known as an on-path attack, is used to describe when an attacker intercepts a channel of communication between two users thereby enabling the monitoring or altering of the transmitted data as well as the opportunity to impersonate either participant. An MitM attack has two phases: interception and decryption. First the attacker intercepts the transmission before it has connected to its destination. A common point of interception is through a free public WiFi hotspot which the user connects to, unsuspectingly also giving the attacker access to their future data exchange. Next the attacker must decrypt any two-way SSL traffic: a common way of doing this without alerting the user is via HTTPS spoofing a tactic where the user is redirected to a fake domain closely impersonating the real target website. MitM attacks can be prevented through the use of encryption protocols such as HTTPS, using secure communication channels, and performing regular software updates.

Phishing




Source: https://www.inside.unsw.edu.au/campus-life/cyber-savvy-staff-can-protect-against-phishing-threats 		Often in the form of an email, phone call or text-message phishing describes a form of fraudulent communication in which an attacker poses as a legitimate person or organisation to lure individuals into providing sensitive data such as banking details, forms of identification or other personal information. The most common and successful form of this type of cyber-attack is spear-phishing: emails or text messages that target the recipient specifically usually by their name. With the advancement of AI, attackers can send fraudulent phone calls or voice messages to trick victims into providing sensitive information, a technique known as voice phishing. The common characteristics of phishing messages include a sense of urgency, hyperlinks and attachments, are from an unexpected source, and offer improbably good deals. Phishing attacks can be prevented through user education and awareness, installing spam detection systems, and utilising multi-factor authentication to verify the identity of users.

Clickjacking




Source: https://en.wikipedia.org/wiki/Clickjacking 		Clickjacking otherwise known as a User Interface (UI) redress attack, is when a user is tricked into unknowingly clicking a malicious element on a webpage often hidden by multiple transparent layers. Once the hidden UI has been clicked malware could be downloaded and malicious web pages be presented requesting sensitive information. Clickjacking attacks come in several different forms including Drag and Drop whereby the attacker captures inputted information from the fill-out fields of a web form. Another type of clickjacking is filejacking whereby the attacker can gain access to the user’s files through the “choose folder” dialog box when downloading a file. Clickjacking can be prevented through the use of frame-busting scripts, utilising X-Frame-Options HTTP headers to restrict framing, and ensuring proper validation of user input to prevent unauthorised interactions with obscured elements.

File Inclusion Attacks




Source: https://knowledge-base.secureflag.com/vulnerabilities/inadequate_input_validation/file_inclusion_vulnerability.html 		File Inclusion attacks predominately target web applications that rely on a scripting run time: a guide for how the mechanics of an application should work such as how the memory is accessed and the appropriate syntax. This type of cyberattack comes in two main forms: remote-, and local-file inclusion. Local file inclusion occurs when the attacker manipulates input parameters forcing the web application to execute files residing on the same server where it can run malicious code or expose sensitive information. Conversely, Remote File Inclusion (RFI) attacks occur attackers include and execute files hosted on a remote server controlled by the attacker, subsequently exposing the system to unauthorised access or the execution of malicious code. File Inclusion Attacks can be prevented by not incorporating user input directly into file inclusion operations, validating input and encoding output thoroughly, and restricting file access permissions thus preventing the unauthorised inclusion of sensitive files.

Server-Side Request Forgery (SSRF)




Source: https://www.geeksforgeeks.org/server-side-request-forgery-ssrf-in-depth/ 		Server-Side Request Forgery (SSRF) is a tactic employed by attackers to manipulates a web application's server-side functions enabling them access or modify data as well as the ability to make unauthorised requests to an unintended location. SSRF attacks can categorised into 3 types according to how the server responds to the initial request. The first is a blind SSRF: this is where the attacker manipulates the server to make a request on behalf of the user without receiving immediate feedback about the outcomes of those requests. The next type is Semi-Blind SSRF and like the Blind SSRF, the attacker also does not receive direct feedback from the server about the outcome of the requests, however through a series of subtle indicators such as differences in response times or the presence of error messages, the attackers can indirectly infer the results. Lastly, there is Non-Blind SSRF where the attacker can immediately observe the effects of their malicious attacks without any limitations. SSRF can be avoided by validating the user's input, implementing a library of allowed URL patterns, and restricting access to sensitive internal resources thus reducing the risk of unauthorised requests being made by the server.

XML External Entity (XXE) Injection




Source: https://www.acunetix.com/blog/articles/xml-external-entity-xxe-vulnerabilities/ 		XML External Entity Injection often abbreviated to XXE Injection, describes when an attacker interferes with an application's processing of XML data. An XXE Injection is so nefarious as not only does it enable the attacker to view files on the application server, but it can also interact with any back end or external systems the application connects and by way of an SSRF attack, compromise the server or other back-end infrastructure. Vulnerabilities that can lead to XXE attacks include a lack of input validation, insecure XML processing libraries and poorly configured systems. There are several different types of XXE injection including: Basic where the attacker inserts a malicious external entity declaration directly into the XML document; Parameter XXE Injection where the harmful code is inserted into the XML document's reusable components; and Entity Expansion XXE Injection where the attacker overwhelms the system with a large number of nested entity references essentially conquering by way of a denial-of-service (DoS) attack. The risk of XXE injection can be reduced by disabling external entity processing and using secure XML libraries that do not expand external entities by default.
